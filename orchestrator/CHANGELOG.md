# Orchestrator workflow changelog

`WORKFLOW_VERSION` (in `orchestrator/workflow.py`) is the version of the
LangGraph `@entrypoint` **body**. It is bumped on INCOMPATIBLE changes to that
body ‚Äî reordered/removed tasks, new required tasks, or changed control flow that
a half-finished checkpoint can't safely resume into. Pure additive changes (a new
optional gate, a new trailing task that legacy checkpoints simply haven't reached)
don't strictly require a bump.

On every resume, the version stored at run creation is compared against the live
constant; a mismatch refuses the resume with a clear error rather than risking a
confusing deserialization failure mid-run.

This file is the human history behind each bump. The deep design notes for each
phase live in `.misc_notes/Done/`.

| Version | Phase | Change |
|---|---|---|
| 2.6.0 ‚Üí 2.7.0 | 77b | **Test-author evidence folder** (second slice of the Phase 77 TDD-evidence layer). The flat `test-author-<id>.md` is replaced by a per-task `task-NN-<id>/test-author/` folder, written ONCE after the authoring process converges (critic satisfied + any re-author done): for a TESTABLE task it copies the FINAL accepted `test_paths` file(s) verbatim (any language/layout, preserving relative paths), plus `results-test-run.md` (the COMPLETE RED run via 77a's `full_output`, proof the suite actually ran red ‚Äî not just the failing delta), `test-snapshot-hash.md` (the deterministic `_hash_test_paths` baseline the diff-gate freezes), and `summary.md` (testable verdict, coverage-critic verdict(s), # re-author rounds, red-review outcome). EVERY TDD task still gets a folder (Phase 73 surfacing): an untestable / degraded task gets `summary.md` only. `TestAuthorResult` gains a `full_run` field (the complete RED run; `red_output` stays the failures-only summary that feeds the implementer's first attempt + the red-review display) ‚Äî a checkpointed shape change on a serde-allowlisted model, so a half-finished 2.6.0 TDD run can't resume across the upgrade; hence the bump. Landmine fixed: `_hash_test_paths` (and the new copier) now resolve `test_paths` through `paths.iter_test_files`, which excludes the orchestrator's own `.orchestrator/` workspace ‚Äî without it the verbatim copies written there would be swept back into the freeze hash by `**` and break the diff-gate. The post-branch task graph is unchanged (the writer is a side-effect after the authored verdict, like its predecessor) and TDD-off runs are byte-for-byte identical. |
| 2.5.0 ‚Üí 2.6.0 | 77a | **Evidence foundations** (generic plumbing for the Phase 77 TDD-evidence layer). Two additive, behaviour-preserving seams, both inert until a consumer is wired in 77c: (1) `StepResult` gains `full_output` ‚Äî the COMPLETE, untruncated runner log (both streams), so the evidence layer can persist the whole RED/attempt run; `detail` stays the short tail/failure report it always was. `_run_script_sync` now fills it on the success and gate-fail paths. (2) `run_retry_block` gains an `on_attempt(attempt, passed, gate_results)` hook that fires ONCE per attempt after the gates resolve, for EVERY attempt ‚Äî crucially the passing (GREEN) one, which `on_gate_failed` never sees and the old `for‚Ä¶else` returned on immediately; it fires before `on_gate_failed` so an attempt's evidence is recorded before any human decision. The post-branch task graph is unchanged and with no hook/consumer the behaviour is byte-for-byte identical ‚Äî but a checkpointed `StepResult` shape changed, so a half-finished 2.5.0 TDD run can't resume across the upgrade and needs a fresh start; hence the bump. |
| 2.4.0 ‚Üí 2.5.0 | 76 | **Autonomous-mode TDD** (re-enable the guard). `tdd = true` + `fully_autonomous = true` is no longer refused at load; instead the per-task station runs a distinct `_run_autonomous_tdd_task` when both are on. With no human red-review/re-author guard, two pieces of machinery stand in: (1) **red-confirm is a HARD gate** ‚Äî a born-green / non-green-baseline / no-script-gate verdict (`TestAuthorResult.degrade_kind` ‚àà the red-confirm failures) aborts the task (`BuildFailed` ‚Üí `status="failed"`) instead of silently degrading; a genuinely-untestable verdict (the author's own judgement) still degrades to classic implement‚Üíqa + a manual check; (2) **bounded re-author** ‚Äî the implement build runs with a BOUNDED budget and `on_exhausted="abort"`, and an exhausted build automatically re-authors the (possibly wrong) tests, capped at `tdd_autonomous_reauthor_max` (default 2), then aborts with a clear reason rather than looping a wrong frozen test to the `autonomous_max_*` safety ceiling. New reachable control flow + per-round `attempt` discriminators on `test_author_task`/`critic_task` ‚Üí a half-finished 2.4.0 checkpoint can't safely resume into it. The supervised / non-TDD per-task graph is byte-for-byte unchanged; `degrade_kind` is an additive optional field, so no serde-allowlist change. `_author_with_review` now returns `(verdict, attempt)` and seeds from a caller-supplied `feedback`/`attempt` so the autonomous cycle can drive repeated rounds with a monotonic counter. |
| 2.3.0 ‚Üí 2.4.0 | 74 | Per-task **coverage critic** (test-meaningfulness backstop). `_author_with_review` now runs a new checkpointed `critic_task` (a separate read-only `coverage-critic` agent) on the authored tests; a `meaningful=False` verdict re-authors via the existing escape (bounded by `tdd_critic_max_attempts`), and if still weak after the budget the run proceeds and records the concern in `manual_checks` (never wedges). A new `@task` in the body's graph ‚Üí a half-finished 2.3.0 checkpoint can't safely resume into it. Gated on `config.tdd_coverage_critic` (default on); `CoverageCriticResult` on the serde allowlist. Generic (a separate agent, not an augmentation of the project-specific `qa.md`). |
| 2.2.0 ‚Üí 2.3.0 | 72b | Supervised **red-review pause + re-author escape**. The per-task station now runs `_author_with_review`: after the green‚Üíred confirm it emits a new `red_review` interrupt (default-on via `tdd_red_review`) for a human to review the failing tests ‚Äî `yes` implements, feedback re-authors the tests (re-running `test_author_task`, which gained `feedback`/`attempt`), `abort` stops the run (`BuildFailed` ‚Üí `status="failed"`). New interrupt + re-author control flow the body can reach ‚Üí a half-finished 2.2.0 checkpoint can't safely resume into it. Suppressed when `tdd_red_review=false` (or `fully_autonomous`). Surfaced through the existing `approve_plan`/`resume` MCP contract (no new tool). |
| 2.1.0 ‚Üí 2.2.0 | 72b | `Task.acceptance_criteria` became **required** (was optional, `default=None`) so every vertical slice carries the observable-behaviour spec the test-author writes tests against; `DecompositionResult.schema_version` 1 ‚Üí 2. Because `DecompositionResult` is checkpointed, a pre-2.2.0 checkpoint can hold a task with no criteria that no longer deserializes ‚Äî the version gate refuses that resume cleanly rather than risking a mid-run `ValidationError`. (Earlier 72b slices ‚Äî `test_author_path`, frontmatter model/tools, RED‚Üíimplementer + artifact ‚Äî were additive and needed no bump.) |
| 2.0.0 ‚Üí 2.1.0 | 72 | The per-task station gained a run-once, checkpointed `test_author_task` before each task's implement loop, gated on `config.tdd` (TDD red-green).
